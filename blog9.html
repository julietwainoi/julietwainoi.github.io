Advanced Terraform Module Usage: Versioning, Nesting, and Reuse Across Environments

On Day 9 of the 30-Day Terraform Challenge, I explored more advanced features of Terraform modules. Building on what I learned on Day 8, I focused on versioning modules and structuring them for use across multiple environments.

Why Module Versioning Matters

By versioning modules using Git tags (e.g., v1.0.0), I can safely introduce changes and test them in dev before deploying to prod. This helps ensure reliability, traceability, and rollback capability.

How I Set It Up

I refactored my EC2 module and hosted it in its own GitHub repository. Then, I used that module in my environment projects like this: 

module "ec2_instance" {
  source  = "git::https://github.com/myusername/terraform-aws-ec2-instance-module.git?ref=v1.0.0"
  ami     = var.ami
  instance_type = var.instance_type
  tags    = var.tags
}

Each environment folder (e.g., dev/, staging/, prod/) had its own main.tf using the appropriate module version.

S3 Backend with Versioning

I also enabled versioning in the S3 bucket used for storing Terraform state: 

resource "aws_s3_bucket" "tf_state" {
  bucket = "my-tfstate-bucket"
  acl    = "private"

  versioning {
    enabled = true
  }

  tags = {
    Name        = "Terraform State Bucket"
    Environment = "global"
  }
}
